# Claude Code Mux 设计原则

Claude Code Mux 的设计哲学是 **"简化复杂的 AI 路由"**。

我们的目标是最大程度降低开发者在同时使用多个 AI 提供商时的复杂度，让配置、监控和管理的每个方面都变得直观。

---

## 1. 我们的设计哲学

### 简洁至上
- 对用户隐藏复杂的 AI 路由逻辑，仅暴露必要的配置。
- 在 CLI 和 Admin UI 中均追求"到达期望结果的最少步骤"。
- 技术复杂度留在内部，用户体验保持简洁。

### 开发者优先体验
- 使用开发者熟悉的术语和结构。
- 配置文件 (TOML) 与 UI 保持相同的思维模型。
- 错误消息清晰传达"出了什么问题"以及"如何修复"。

### 性能即用户体验
- 约 5MB 内存占用、<1ms 路由开销不仅仅是性能指标——它们就是用户体验。
- 快速响应建立信任，轻量资源使用简化部署。

---

## 2. Admin UI 产品原则

### 价值优先，成本次之
**先向用户展示配置的价值，再把复杂设置放到后面。**

#### 示例
- ❌ 错误：添加 Provider 时，在一个屏幕上立即要求填写 API Key、Base URL 和类型
- ✅ 正确：逐步引导："要添加哪个 Provider？" → 选择类型 → 命名 → 输入 API Key

#### 检查点
- [ ] 添加新功能时，用户能否先理解"为什么要做这件事"？
- [ ] 复杂选项是否可以折叠到"高级设置"下？
- [ ] 必填项和选填项是否清晰区分？

---

### 易于回答
**用户在每个屏幕上应该能立即明白要选择和输入什么。**

#### 示例
- ❌ 错误：抽象标签如"模型映射配置"
- ✅ 正确："选择 Provider" → "实际模型名称" → "选择为 claude-sonnet-4 使用哪个 Provider"

#### 检查点
- [ ] 每个输入字段是否有占位文本或帮助文本？
- [ ] 选项（单选、下拉）是否限制在 3-5 个以内？
- [ ] 错误消息是否解释了"出了什么问题以及如何修复"？

---

### 一页一事
**每个屏幕只处理一个目的。**

#### 示例
- ✅ 概览页：仅查看整体状态
- ✅ Provider 管理页：仅查看、添加、编辑、删除 Provider
- ✅ 模型管理页：仅查看、添加、编辑、删除 Model
- ✅ 路由设置页：仅配置路由规则

#### 检查点
- [ ] 能否用一句话描述该页面的核心目的？
- [ ] 页面标题 (h1) 是否准确反映了用户意图？
- [ ] 页面上的多个功能是否互不冲突？

---

## 3. UX 写作原则

### 简洁 = 去除无意义的词语
- ❌ "我们正在向您展示一个已配置的 Provider 列表"
- ✅ "Provider 列表"

### 专业但不晦涩
- ✅ "Provider" - 开发者熟悉的术语
- ✅ "Model mapping" - 含义清晰
- ❌ "推理端点配置" - 不必要地复杂

### 面向操作的文字
- ✅ "添加 Provider"
- ✅ "编辑 Model"
- ❌ "添加"（不清楚添加的是什么）

---

## 4. 指标与监控

### 性能指标作为 UX 反馈
- 在 Admin UI 中实时显示路由开销和内存使用情况
- 用户可以立即看到"我的配置有多高效"

### 错误追踪
- 追踪每个 Provider 的错误率和响应时间
- 立即识别并响应有问题的 Provider

### 检查点
- [ ] 重要指标是否以数字形式展示？
- [ ] 是否可以通过红色/警告图标立即注意到异常情况？
- [ ] 是否可按时间段和 Provider 查看统计数据？

---

## 5. 无障碍与通用 UX

### 键盘导航
- 所有功能必须可仅通过键盘访问
- Tab 顺序必须是逻辑合理的
- 焦点状态必须清晰可见

### 语义化 HTML
- 按钮是 `<button>`，链接是 `<a>`，表单是 `<form>`
- 屏幕阅读器必须能理解页面结构

### 检查点
- [ ] 是否所有任务都可以仅用 Tab + Enter 完成，无需鼠标？
- [ ] 信息是否不仅仅通过颜色传递？（色盲支持）
- [ ] 错误消息是否可被屏幕阅读器读取？

---

## 6. 交互与反馈

### 即时反馈
- 按钮点击 → 立即显示加载状态
- 表单提交 → 立即显示成功/失败通知
- 配置变更 → 显示"未保存的更改"指示

### 渐进式披露
- 先显示基本设置，通过"高级设置"按钮展开其余内容
- Provider 添加步骤逐步推进，且支持返回

### 检查点
- [ ] 每个操作是否有即时反馈？
- [ ] 加载过程中用户是否无需空等？
- [ ] 破坏性操作是否提供了撤销/取消功能？

---

## 7. 工具与自动化

### 配置验证
- 保存前验证 TOML 配置文件
- 无效配置无法保存，并给出清晰错误消息

### 智能默认值
- 添加新 Provider 时提供合理的默认值
- 自动分配 Model mapping 优先级为 1, 2, 3...

### 开发效率
- 通过热重载测试配置变更，无需重启服务器
- 配置变更后，从 Admin UI 一键"保存并重启"

---

## 8. 开发者实用检查清单

### 添加新功能前
- [ ] 该功能是否遵循一页一目的设计？
- [ ] 是否先向用户展示价值？
- [ ] 问题和选项是否清晰明确？

### 编写 UI 组件时
- [ ] 是否去除了所有文本中无意义的词语？
- [ ] 是否仅用键盘即可操作？
- [ ] 错误和加载状态是否清晰？

### 设计表单时
- [ ] 必填字段是否清晰标注？
- [ ] 每个字段是否有占位/帮助文本？
- [ ] 校验检查（重复、格式）是否即时进行？

### 处理错误时
- [ ] 错误消息是否解释了"什么"出了问题？
- [ ] 错误消息是否解释了"如何"修复？
- [ ] 用户是否能看到该错误？

### 优化性能时
- [ ] 页面是否在 2 秒内加载完成？
- [ ] 是否对用户操作立即响应？
- [ ] 是否已消除不必要的网络请求？

## 示例：Provider 添加流程

### Before（❌ 所有内容挤在一个屏幕）
```
添加 Provider
- 名称: [        ]
- 类型: [选择▼]
- API Key: [        ]
- Base URL: [        ]
- 模型列表: [        ]
- 启用: [v]
[添加]
```

### After（✅ 分步引导）
```
第 1 步：您想添加哪个 Provider？
○ Anthropic  ○ OpenAI  ○ OpenRouter
（大卡片格式供选择）

第 2 步：为您的 Provider 命名
名称: [anthropic-main]
（此名称将用于在 Model mappings 中引用它）

第 3 步：输入您的 API Key
API Key: [sk-ant-...]
（您可以从该提供商的网站获取）

第 4 步：完成！
✓ anthropic-main Provider 已添加
现在转到 Model 标签页，使用此 Provider 添加模型
```
